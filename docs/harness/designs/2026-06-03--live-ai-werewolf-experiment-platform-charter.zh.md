# Live AI 狼人杀实验平台宪章

日期：2026-06-03
状态：草案

本文档不是一个 PR 的实现计划。它是将 Werewolf-agent 从 provider 驱动的运行时基础，转型为客户端无关的实时 AI 狼人杀实验平台的产品路线与系统架构宪章。

## 1. 产品北极星

Werewolf-agent 应成为一个可配置、可观察、实时运行的 AI 狼人杀对战平台。

该平台不仅仅是"能运行一局游戏"。完整的产品体验应让用户能够：

1. 开启一局新的 AI-vs-AI 狼人杀对战，
2. 配置角色、座位、provider、模型、提示词和策略，
3. 实时运行比赛，
4. 通过可信的视角观察比赛，
5. 审查每项操作背后的事件、快照、清单、provider 和失败审计证据，
6. 保存并重播已完成的比赛，
7. 比较多次运行，并后续评估模型 / 提示词 / 角色表现。

用户界面的感觉应更接近游戏大厅和比赛驾驶舱。底层的系统仍然是一个实验平台，包含结构化配置文件、清单、事件流、快照、审计产物和可重现的运行包。

## 2. 现有基础

G1a-G1h 建立了实时运行时基础：

- 确定性路径和 provider 驱动的游戏路径，
- 最小狼人杀游戏引擎，
- mock agent 合约，
- 狼人共识和失败恢复，
- 假 provider 和 DeepSeek provider 烟雾测试路径，
- provider 追踪和失败审计输出面，
- 离线 provider 重播 HTML 产物，
- 包含 `events.jsonl`、运行时快照、提示词清单、provider 生命周期事件和标准日志包兼容性的实时运行时事件脊柱。

当前路线对齐风险：

- 最近的 GitHub 历史显示 G1h 实现已合并，
- 部分路线/状态文档可能仍将 G1h 描述为 `next_candidate`，
- 这种过时状态的清理是路线收尾任务，而非新的产品功能。

## 3. 体验架构

主要的用户体验应围绕比赛展开，而非原始文件或 JSON。

### 3.1 首页 / 大厅

首个屏幕应提供：

- 开始新比赛，
- 比赛历史，
- 最近或正在运行的比赛状态，
- 配置文件、provider 和产物的次级入口。

首个产品操作是游戏化的：开始一局新的 AI-vs-AI 狼人杀比赛，或打开一局历史比赛。

### 3.2 开始新比赛

开始新比赛应打开一个游戏化的角色配置界面，而非普通的设置表单。

推荐的交互方式：

1. 用户看到带有角色插图和角色名称的角色卡片，
2. 点击角色卡片，所选卡片动画进入左侧聚焦位置，
3. 其他角色卡片淡出、折叠或移出焦点，
4. 右侧配置面板无缝出现在所选角色旁，
5. 用户可以接受默认设置、选择 provider/模型/AI 策略、编辑提示词设置，并保存本地可重用配置，
6. 用户之后可以导入已保存的本地配置。

UI 使用游戏语言表达：

- 开始新比赛，
- 比赛历史，
- 房间配置，
- 角色卡，
- 玩家座位，
- AI 阵容，
- 开始比赛，
- 重播。

系统和产物使用工程语言表达：

- 配置文件，
- run_id，
- experiment_id，
- 提示词清单，
- 事件脊柱，
- 快照，
- provider 追踪，
- 失败审计。

### 3.3 角色默认值与座位覆写

AI 配置以角色默认值为优先，座位可覆写。

默认层：

- 所有狼人继承狼人默认 AI 配置，
- 所有村民继承村民默认 AI 配置，
- 预言家继承预言家默认 AI 配置，
- 女巫继承女巫默认 AI 配置。

覆写层：

- 任何单个座位可以覆盖 provider、模型、提示词配置文件、策略、温度、超时或预算守卫，
- 座位覆写在设置 UI 中可见，
- 最终运行时配置将每个座位解析为一份完整的独立 AI 配置。

概念数据结构：

```text
role_defaults
  -> seat_overrides
  -> resolved_seat_configs
```

运行时应消费已解析的座位配置。UI 和本地保存的配置文件模板可以保留角色默认值和座位覆写，以提高可用性。

### 3.4 预检

在实时运行启动前，平台应执行预检：

- 配置文件模式验证，
- provider 可用性，
- 所需密钥存在但不泄露密钥值，
- 预算守卫，
- 提示词清单预览，
- 可见性设置，
- 输出产物目标路径，
- 运行模式：实时或重播。

预检应成为产品体验的一部分，而不仅仅是 CLI 验证步骤。

### 3.5 实时比赛驾驶舱

驾驶舱应让用户无需打开原始 JSON 就能理解比赛：

- 当前白天/夜晚/阶段/轮次，
- 玩家座位、角色、存活/死亡状态，
- 公开事件流，
- 当前行动者和最新操作，
- 投票和夜间行动摘要，
- provider 状态、失败、延迟和预算指标，
- 指向清单、provider 追踪、失败审计、快照和最终日志的审计链接，
- 视角切换器：上帝 / 公开 / 角色 / 团队。

### 3.6 比赛历史和重播

历史比赛应支持：

- 列出已完成的运行，
- 通过同一观察者模型打开重播（尽可能），
- 查看所使用的配置文件，
- 打开最终日志和审计产物，
- 后续通过实验组比较多次运行。

## 4. 系统架构

产品应保持客户端无关的架构：

```text
配置 / 比赛设置
-> Python 运行时游戏引擎 + agent/provider 循环
-> 事件脊柱 + 快照 + 提示词清单 + provider 生命周期
-> 本地观察者服务器
-> Qt/QML 或 Web 观察者客户端
-> 重播 / 审计 / 导出
-> 实验编排
-> 评估平台
```

核心组件：

- Python 运行时：拥有游戏引擎、agent 循环、provider 适配器、事件总线、日志写入器、快照、清单和最终包生成。
- 事件脊柱：暴露稳定的运行时事件，包含指向快照、提示词清单条目、provider 追踪、失败审计和最终日志的引用。
- 观察者服务器：暴露本地 REST 加流式协议，用于运行控制、事件观察、快照访问、产物查询和历史运行重播。
- 观察者客户端：仅消费客户端无关的协议；Qt/QML 是推荐的首个富客户端，但协议也必须支持未来的 Web 客户端。
- 配置文件注册表：存储本地可重用的比赛/配置文件模板，并将角色默认值和座位覆写解析为每个座位的运行时配置。
- 实验编排：对运行进行分组，支持重播/实时双模式，记录配置差异和运行元数据。
- 评估层：后续将实际运行数据转换为评分卡、聚合、样本量感知比较和排行榜。

## 5. 阶段划分

### 阶段 A：平台宪章与体验架构

目标：定义产品路线、系统架构、体验模型、信任合约、阶段退出条件和反收缩门禁。

输入：

- 现有产品文档，
- 现有路线图，
- 已接受的客户端无关观察者 ADR，
- 现有 G1h 基础。

输出：

- 本宪章，
- 后续由已批准决策衍生的路线文档更新。

退出演示：

- 不需要运行时演示，
- 每个后续实现计划可以引用本文档来确定产品适配性、范围边界和收缩风险。

边界：

- 不写运行时代码，
- 不实现观察者服务器，
- 不实现 Qt/Web，
- 不进行评分或排行榜工作。

### 阶段 B：G1h 合约收尾 + 路线文档对齐

目标：使规范文档与已合并的 G1h 实况对齐，移除过时的路线状态。

输出：

- README / PRODUCT_ONE_PAGER / ROADMAP / TASKS 状态对齐，
- 明确声明 G1h 事件脊柱是 G2a+ 的基础，
- 不虚假声称存在观察者服务器、富客户端、配置文件编辑器、竞技场或排行榜。

退出演示：

- 路线文档不再显示已合并的 G1h 工作的过时 `next_candidate` 实况，
- 下一个实现线路明确为 G2a 本地观察者服务器。

边界：

- 仅限文档/状态对齐，除非有受约束的实现计划另有说明。

### 阶段 C：G2a 本地观察者服务器 / 协议控制平面

目标：通过本地客户端无关协议暴露运行时和历史运行数据。

输出：

- 用于运行、状态、快照、产物、清单、provider 追踪和失败审计的 REST 端点，
- 用于运行时事件和状态变化的流式端点，
- 用于启动、停止和检查的运行控制面（计划允许范围内），
- 用于启动默认 6 人对局的最小 match/profile 合约种子，但不做完整配置编辑器，
- 与已完成运行重播和实时运行观察的兼容性。

退出演示：

- 同一个运行可以通过协议调用进行观察和审计，无需手动打开本地 JSON 文件，
- 流式消费者可以从事件和快照跟踪比赛进展。

边界：

- 没有富客户端，
- 没有提示词编辑器 UI，
- 没有多 provider 竞技场，
- 没有排行榜，
- 不允许客户端直接访问 Python 运行时内部。

### 阶段 D：G2b 观察者驾驶舱 MVP

目标：提供首个丰富的比赛观察体验，以 Qt/QML 作为推荐的首个客户端。

起始骨架：

- `clients/qt_observer` 是该阶段的本地 Qt6 Quick 起始工程。
- 当前该骨架仅为自动生成内容，不包含观察者协议集成、比赛驾驶舱 UI、God/Role View、运行控制或历史/重播 UI。

输出：

- 游戏化的比赛驾驶舱，
- 运行状态，
- 玩家面板，
- 实时事件流，
- provider 追踪摘要，
- 审计链接，
- 基本的重播/打开历史运行支持（在协议允许的范围内）。

退出演示：

- 用户可以从 Home 进入 Start New Match，接受默认 6 人配置，查看 preflight 摘要，启动运行，进入实时驾驶舱，并在之后通过历史/重播重新打开，
- 用户无需阅读 JSON 即可观看实时或重播的 AI-vs-AI 比赛，
- 用户能够理解阶段、玩家状态、当前行动者、主要事件和 provider/失败状态。

边界：

- Qt 必须消费 REST/流/事件协议，
- 不直接绑定 Python 运行时对象或私有 API，
- 没有完整的提示词编辑器，
- 没有排行榜。

### 阶段 E：可见性信任层

定位：横切门禁 + 独立强化里程碑。

横切门禁：

- G2a 协议必须携带可见性字段、投影引用和权限边界，
- G2b 驾驶舱必须分别渲染上帝 / 公开 / 角色 / 团队视角，
- G2c/G2d 配置文件和提示词系统必须声明角色信息、可见性和提示词清单条目如何影响运行。

独立强化里程碑：

- 正式的上帝/公开/角色/团队投影合约，
- 隐藏信息泄露测试，
- 审计证据，说明为什么每个视角能看到或不能看到某个事件或字段，
- UI 可见性徽章和视角警告。

退出演示：

- 相同的底层事件在上帝、公开、角色和团队视角下呈现不同，
- 隐藏信息泄露由测试或显式验证产物覆盖，
- UI 可以显示数据被隐藏的原因，而非静默省略。

边界：

- 不削弱狼人杀信息不对称，
- 不将上帝视角和角色视角合并为一个便利载荷，
- 不存在将隐藏角色泄露到角色受限视图中的提示词或 provider 产物。

### 阶段 F：提示词/配置 MVP

目标：使比赛配置可重用、可验证、可审计且面向用户。

输出：

- YAML 或本地配置文件模板格式，
- 角色默认值，
- 座位覆写，
- 已解析的座位配置，
- 配置文件验证器，
- 提示词清单生成，
- 本地保存/导入可重用配置，
- 用于选择 provider/模型/提示词/策略的设置 UI 入口。

退出演示：

- 用户可以从默认配置开始比赛，
- 自定义角色默认值和座位覆写，
- 保存本地配置，
- 之后导入配置，
- 启动运行，其清单追踪每座位的已解析 provider/模型/提示词/策略。

边界：

- 没有托管账户系统，
- 没有云端配置同步，
- 没有完整的多人 provider 竞技场，
- 不允许运行时不可追踪的自由文本提示词变更。

### 阶段 G：实验编排

目标：将单个比赛转变为可重复的实验。

输出：

- experiment_id 和运行分组，
- 基于同一观察者模型的重播/实时双模式，
- 配置差异追踪，
- 批次元数据，
- 成本/失败/延迟摘要，
- 可比较的导出。

退出演示：

- 同一个配置组可以运行多次比赛，
- 用户可以比较运行元数据、失败、成本、配置差异和结果摘要。

边界：

- 没有虚假排行榜，
- 没有基于真实数据和受约束评估计划的评分公式变更，
- 没有不受控制的 provider 成本扩张。

### 阶段 H：评估平台

目标：将实际运行输出转换为稳健的评估产品。

输出：

- 单次运行评分卡，
- 多次运行聚合，
- 按角色分离的指标，
- 样本量警告，
- provider/模型/版本比较，
- 可导出的评估报告，
- 真实排行榜。

退出演示：

- 评估结果基于足够的实际运行次数，
- 每个分数都可追溯到运行产物，
- 排行榜条目包含样本量和来源警告。

边界：

- 在 G1h/G2/G3 数据稳定且充足之前，不发布真实排行榜，
- 不将人工金牌样本排行榜呈现为真实数据，
- 不允许无法追溯到已接受的评估设计的评分变更。

## 6. 横切信任合约

### 可见性合约

每个事件、快照字段、提示词引用和客户端载荷必须声明或推导其可见性边界。

必需的视角：

- 上帝，
- 公开，
- 角色，
- 团队。

上帝视角用于审计。角色视角用于证明 agent 或观察者仅能看到允许的信息。团队视角在狼人共识中尤为重要。

### 产物溯源

所有用户可见的主张应可追溯到结构化产物：

- 事件 ID，
- 快照 ID，
- 提示词清单引用，
- provider 追踪引用，
- 失败审计引用，
- 最终日志引用。

### 密钥脱敏

密钥不得进入：

- 运行时事件，
- 快照，
- 提示词清单，
- 用于用户展示的 provider 追踪载荷，
- 重播产物，
- 保存的本地配置文件。

配置文件可以通过逻辑名称引用 provider 名称和本地密钥标识，但不得嵌入密钥值。

### 失败审计

Provider 失败、解析失败、无效操作、超时和恢复路径必须保持可见和可审计。

平台不得将 provider 失败静默修复为表面有效的行为。

### 客户端无关协议

客户端消费协议载荷，而非 Python 内部结构。

Qt/QML 可以作为首个富客户端。Web 可以后续跟进。两者都必须能够消费相同的概念性观察者模型。

### 配置清单

每次运行应生成一份清单，说明：

- 所选比赛模板，
- 角色默认值，
- 座位覆写，
- 已解析的座位配置，
- provider/模型/提示词/策略引用，
- 运行时选项，
- 可见性设置，
- 脱敏保证。

## 7. 退出演示要求

每个阶段都需要一个人类可见的退出演示。

这防止路线变成不可见的基础设施：

- 阶段 A：已批准宪章，未来计划可以引用。
- 阶段 B：路线文档与已合并的 G1h 实况对齐。
- 阶段 C：无需手动检查 JSON 即可通过协议观察运行。
- 阶段 D：无需阅读 JSON 即可在驾驶舱中观察比赛。
- 阶段 E：按视角渲染，具有防泄露证据。
- 阶段 F：保存/导入的本地配置文件启动可追踪的运行。
- 阶段 G：运行组比较，包含配置差异和失败/成本元数据。
- 阶段 H：带来源链接和样本量警告的评估输出。

## 8. 反收缩门禁

路线不得收缩为以下任何形态：

- 以静态 HTML 重播作为主要用户体验，
- 用 JSON 查看器代替观察者驾驶舱，
- 以单局烟雾测试代替可配置的比赛平台，
- Qt 客户端直接绑定到 Python 运行时内部，
- 将提示词编辑作为不可追踪的文本框，
- 角色视图静默泄露隐藏信息，
- 将 provider 失败隐藏为有效游戏行为，
- 在拥有足够的实际运行数据之前发布虚假排行榜，
- 在评估平台拥有实际证据之前更改评分公式。

## 9. 开放设计问题

1. 首个富客户端应保持 Qt/QML 优先、Web 后续，还是 Web 应成为同等的早期目标？
2. G2a 应使用 REST + SSE、REST + WebSocket，还是同时支持两者并采用同一规范事件模型？
3. 驾驶舱默认应暴露多少原始 provider 响应或推理过程？
4. 原始 provider 响应应仅存在于受控的审计产物中，而 UI 展示脱敏摘要？
5. 第一个配置 MVP 应包含多少提示词编辑功能，与后续的提示词编辑器工作如何划分？
6. 本地保存的配置文件应如何存储、命名、版本化和迁移？
7. 即使人-vs-AI UI 在当前范围之外，协议中现在是否应为人类玩家座位预留空间？
8. 在任何 G4 排行榜视图可以自称为"真实"之前，最少需要多少次实际运行？

## 10. 推荐的下一个实现线路

推荐顺序：

1. 阶段 B 路线收尾：使文档与已合并的 G1h 实况对齐。
2. 阶段 C 实现计划：G2a 本地观察者服务器 / 协议控制平面。
3. 阶段 C 必须播下默认模板开局所需的最小 match/profile 合约种子，但不能变成完整配置编辑器。
4. 阶段 E 可见性合约切片，从一开始嵌入 G2a。
5. 阶段 D 观察者驾驶舱 MVP，在协议合约建立之后，并包含默认模板开局流程，而不是只观察已有运行。
6. 阶段 F 提示词/配置 MVP，在比赛设置和运行启动边界明确之后。

不要直接从此宪章跳转到 Qt 页面或单个服务器端点。下一个实现计划应证明观察者协议能够支持游戏化的比赛体验、可见性信任边界、产物溯源以及后续的配置/实验层。
